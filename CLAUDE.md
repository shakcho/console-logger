## Testing rules

**Always write or update tests when changing code.** Tests live in `src/__tests__/` and mirror the source structure:

```
src/__tests__/
  CircularBuffer.test.ts
  levels.test.ts
  formatter.test.ts
  Konsole.test.ts
  transports/
    ConsoleTransport.test.ts
    HttpTransport.test.ts
    StreamTransport.test.ts
    FileTransport.test.ts
```

Guidelines:
- Use `SpyTransport` (defined inline in `Konsole.test.ts`) to capture entries without real I/O when testing `Konsole` logic.
- `NO_COLOR=1` is set globally in `vitest.config.ts` — assertions on formatted output will not contain ANSI codes; use `stripAnsi()` only when mocking `process.stdout.write` directly.
- Error and fatal levels route to `process.stderr`; all others route to `process.stdout`. Spy the right stream.
- `FileTransport` tests should create temp files in `os.tmpdir()` and clean them up in `afterEach`.
- When testing `HttpTransport`, pass `fetchImpl` explicitly (a `vi.fn()` mock) rather than relying on `globalThis.fetch`.
- Run `npm run test:run` before committing. All tests must pass.

## Environment requirements

- **Node.js ≥ 18** is required for server-side use (native `fetch`). Older versions must pass a `fetchImpl` option to `TransportConfig`.
- `useWorker: true` is supported on both browser (Web Worker) and Node.js (`worker_threads`). Falls back gracefully if neither is available.
- The global print flag (`__KonsolePrintEnabled__`) is stored on `globalThis` — works in both environments.

## Architecture

This is a zero-dependency TypeScript library (`konsole-logger`) for namespaced browser/Node.js logging. The entry point is `src/index.ts`.

### Log levels

`trace=10 → debug=20 → info=30 → warn=40 → error=50 → fatal=60`. Defined in `src/levels.ts`. The `level` constructor option sets the minimum threshold; entries below it are discarded before being stored or forwarded to transports. `log()` is an alias for `info()`.

### Output format selection

`format: 'auto'` (default) selects at construction time via `createAutoFormatter()` in `src/formatter.ts`:
- `isBrowser` → `BrowserFormatter` (CSS `%c` badges in DevTools)
- `isNode && isTTY()` → `PrettyFormatter` (ANSI colors, respects `NO_COLOR` / `FORCE_COLOR`)
- `isNode && !isTTY()` → `JsonFormatter` (newline-delimited JSON to stdout/stderr)

`USE_COLORS` is computed once at module load time (not per call). Errors and fatals always go to `stderr`; all other levels go to `stdout`.

Available formats: `'auto' | 'pretty' | 'json' | 'text' | 'browser' | 'silent'`.

### Timestamp system

Timestamps are configurable via the `timestamp` option on `KonsoleOptions` (and `KonsoleChildOptions`). The option accepts either a `TimestampFormat` preset string, a custom `(date, hrTime?) => string` function, or a full `TimestampOptions` object with `{ format, highResolution }`.

Available presets: `'datetime'` (default for pretty/text/browser), `'iso'` (default for json), `'time'`, `'date'`, `'unix'`, `'unixMs'`, `'none'`.

Key implementation details:
- `resolveTimestampConfig()` and `formatTimestamp()` in `src/formatter.ts` are the single source of truth for all timestamp formatting across formatters and transports.
- Each formatter class (`PrettyFormatter`, `JsonFormatter`, `TextFormatter`, `BrowserFormatter`) accepts `FormatterOptions` with an optional `timestampFormat` field.
- `createFormatter()` and `createAutoFormatter()` accept an optional `tsFormat` parameter that is forwarded to the formatter constructor.
- `Konsole` stores `currentFormat`, `timestampFormat`, and `highResolution` on the instance. `setTimestamp()` recreates the formatter.
- `LogEntry.hrTime` (optional `number`) stores a high-resolution monotonic timestamp in nanoseconds when `highResolution: true`. Captured via `getHrTime()` in `src/env.ts` using `process.hrtime.bigint()` (Node) or `performance.now()` (browser).
- `JsonFormatter` includes `hrTime` in JSON output when present on the entry.
- `BrowserFormatter` now renders timestamps (previously omitted, relied on DevTools). Uses `'datetime'` by default.
- `toTextLine()` in `src/transports/base.ts` outputs `YYYY-MM-DD HH:MM:SS.mmm` (was time-only before).
- `exposeToWindow()` exposes `setTimestamp()` on both per-logger and global level via `window.__Konsole`.

### Calling conventions

All four are supported, parsed in `Konsole.parseArgs()`:
```ts
logger.info('message')                          // simple string
logger.info('message', { userId: 1 })           // string + fields object
logger.info({ msg: 'message', userId: 1 })      // Pino-style object-first
logger.error(new Error('oops'))                 // Error → msg=err.message, fields.err=Error
```
Call-site fields are merged with child bindings at log time; call-site fields win on collision.

### Core source files

| File | Purpose |
|------|---------|
| `src/Konsole.ts` | Main class; static instance registry; `child()` factory; `addLog()` pipeline |
| `src/CircularBuffer.ts` | Fixed-capacity ring buffer; auto-evicts oldest entries when full |
| `src/levels.ts` | Level constants, `LEVEL_LABELS` (3-char badges), `isValidLevel()` |
| `src/formatter.ts` | Five formatters + `createFormatter()` / `createAutoFormatter()` factories + `resolveTimestampConfig()` / `formatTimestamp()` |
| `src/env.ts` | `isBrowser`, `isNode`, `isTTY()`, `getGlobalFetch()`, `getHrTime()` — computed at module load |
| `src/types.ts` | All shared types: `LogEntry`, `Transport`, `KonsoleOptions`, `TransportConfig`, `TimestampFormat`, `TimestampOptions`, etc. |
| `src/workerAdapter.ts` | Platform-agnostic worker factory (`KonsoleWorker` interface); Web Worker (browser) or `worker_threads` (Node.js) |
| `src/transports/base.ts` | `toJsonLine()` / `toTextLine()` / `toLine()` shared by file+stream transports |

### Transport system

The `Transport` interface (in `src/types.ts`) requires: `name`, `write(entry)`, `destroy()`, and optional `flush()`. Four implementations live in `src/transports/`:

| Class | File | Notes |
|-------|------|-------|
| `HttpTransport` | `HttpTransport.ts` | Batching HTTP POST with configurable `batchSize`, `flushInterval`, exponential-backoff retry, per-entry `filter`/`transform`. Original `src/Transport.ts` is a deprecated re-export. |
| `ConsoleTransport` | `ConsoleTransport.ts` | Wraps a `Formatter`; useful when the main logger uses `format: 'silent'` but you still want formatted console output from specific transports. |
| `StreamTransport` | `StreamTransport.ts` | Writes JSON/text lines to any `WritableLike` (duck-typed — consumers don't need `@types/node`). Node.js only. |
| `FileTransport` | `FileTransport.ts` | Extends `StreamTransport`. Opens file with dynamic `import('node:fs')`. Buffers entries during async open; call `await ft.ready()` to wait for the file handle before asserting file contents in tests. |

`vite.config.ts` marks all `node:` built-ins as Rollup externals (`external: (id) => id.startsWith('node:')`) so they are never replaced with browser stubs in the bundle.

`addTransport()` and `KonsoleOptions.transports` accept both `Transport` instances and `TransportConfig` plain objects (auto-wrapped in `HttpTransport` via the `isTransportConfig()` type guard inside `Konsole.ts`).

### Child loggers

`logger.child(bindings, options?)` creates a child via the private static `Konsole.createChild()` factory, which uses `Object.create(Konsole.prototype)` to bypass the constructor. Key invariants:

- **Shared**: `logs` buffer, `useWorker` — same reference, mutations visible both ways. `formatter` is shared unless the child overrides `timestamp`.
- **Copied**: `transports` array (new array, same `Transport` instances — `child.addTransport()` won't affect parent), `criteria`, scalar config, `timestampFormat`, `highResolution`
- **Child-own**: `bindings` (accumulated from all ancestor layers via spread), fresh `currentBatchStart`, no `cleanupIntervalId`
- Children are never registered in `Konsole.instances`

### JSON log schema

All structured output follows the Pino-compatible schema:
```json
{ "level": 30, "levelName": "info", "time": "<ISO string>", "namespace": "App", "msg": "...", ...fields }
```

### Worker mode dual-storage

When `useWorker: true`, logs are written to **both** the main-thread `CircularBuffer` (for synchronous `getLogs()`) and posted to the shared worker (for transport processing). The worker contains a self-contained `CircularBuffer` copy as an inline string inside `getWorkerCode()`. `getLogsAsync()` fetches from the worker; `getLogs()` reads the main-thread buffer.

Worker creation is handled by `createPlatformWorker()` in `src/workerAdapter.ts`, which returns a unified `KonsoleWorker` interface. In browsers it uses `Blob` + `URL.createObjectURL` to create a Web Worker; in Node.js it dynamically imports `worker_threads` and prepends a shim that maps `parentPort` onto the `self.onmessage`/`self.postMessage` convention. Messages sent before the async Node.js import resolves are buffered and flushed once the worker is ready.

### Console output control

`criteria` (deprecated) controls whether the formatter prints output. It can be `boolean` or `(entry: LogEntry) => boolean`. The global flag `Konsole.enableGlobalPrint(true)` overrides all per-instance criteria — useful for unsilencing all loggers in a production DevTools session via `Konsole.exposeToWindow()` / `window.__Konsole`.

### Build output

`vite build` produces `dist/konsole.js` (ESM) and `dist/konsole.umd.cjs` (UMD), with bundled type declarations at `dist/index.d.ts`. The library has zero runtime dependencies.
